## 1. 项目目标

`dxm` 是 DXM 大项目 AI 协作 skill 的源码仓。它把“四模式路由、有界 project-grill、持久化项目基线、readiness 审计、证据矩阵、completion receipt、Trellis 大任务治理”沉淀为可安装 skill 与确定性工具，供支持 Agent Skills 的运行时复用。

主要调用方：

- 触发 `/dxm` 或 `/dxm trellis` 的 agent：先读 `skills/dxm/SKILL.md`，锁定 root/mode/scope，再进入 `audit`、`init`、`task` 或 `scaffold-only`。
- 维护者：运行 `scaffold_dxm.py` 写入文档/基线，运行 `validate_dxm.py` 做只读 audit、baseline 或 receipt 校验。

成功产物：

- `skills/dxm/SKILL.md`：运行时触发契约。
- `skills/dxm/scripts/scaffold_dxm.py`：实际 scaffold / baseline hydration / dry-run / refresh / Trellis 初始化实现。
- `skills/dxm/scripts/dxm_contract.py`：baseline、marker、readiness 与 completion receipt 的共享契约。
- `skills/dxm/scripts/validate_dxm.py`：只读 validator CLI。
- `skills/dxm/assets/templates/`：写入用户项目的五份长期文档模板。
- `skills/grilling`、`skills/grill-with-docs`、`skills/grill-me`、`skills/domain-modeling`：可选增强；核心 DXM 不依赖它们也能完成 0–3 有界澄清。

## 2. 运行模式

### 2.1 Skill 触发模式

用户输入 `/dxm`、`/dxm trellis` 或要求生成/刷新 DXM 规范时，agent 读取 `skills/dxm/SKILL.md`：

1. 规范化并锁定项目根目录、模式和允许影响范围；root mismatch 时停止写入。
2. 四选一：`audit` 严格只读；`init` 建基线；`task` 沿用既有基线；`scaffold-only` 只补模板且不宣称 READY。
3. `init` 先从第一性原理和本地证据判断，只单批询问 0–3 个阻塞问题；完整 grilling 仅 explicit opt-in。
4. `init` 落盘 `.dxm/project.json`，再非破坏性 scaffold 并审计 readiness；`task` 不重复初始化。

### 2.2 Scaffold CLI 模式

维护者或 agent 可直接运行：

```bash
python skills/dxm/scripts/scaffold_dxm.py --mode scaffold-only --root <project-root>
```

常用变体：

- `--mode init`：写模式锁；必须同时给出有效 `--baseline`，否则在任何项目写入前 exit 2。
- `--mode scaffold-only`：模板写模式锁；拒绝 baseline，输出 `readiness: NOT_EVALUATED`，不作四态完成判断。
- `--baseline <json>`：校验 root/schema，持久化 `.dxm/project.json`，并把专用 managed block 水合到链路文档。
- `--dry-run`：只报告计划动作，不写文件。
- `--refresh-blocks`：只刷新 DXM/Trellis 管理块，保留人工内容。
- `--inventory-depth N`：控制初始化文件结构种子的目录深度。
- `--force`：覆盖既有 DXM 目标文件；只允许在用户明确接受丢失人工内容风险时使用。
- `--self-test`：对已安装 skill 做轻量冒烟。

scaffold 写入成功不等于 READY。CLI 末尾会打印 `ABSENT` / `PARTIAL` / `READY` / `BROKEN` 审计状态；无 baseline 的兼容 scaffold 通常是 `PARTIAL`。

### 2.3 Validator 模式

只读入口：

```bash
python skills/dxm/scripts/validate_dxm.py audit --root <project-root> [--require-trellis] [--json]
python skills/dxm/scripts/validate_dxm.py baseline --file <baseline.json> [--json]
python skills/dxm/scripts/validate_dxm.py receipt --root <project-root> --file <completion.json> [--json]
```

退出码：`0=READY/valid`、`2=BROKEN/invalid`、`3=PARTIAL`、`4=ABSENT`。validator 不创建目录、不修复文件、不改变运行态。

### 2.4 Trellis 模式

运行：

```bash
python skills/dxm/scripts/scaffold_dxm.py --mode init --root <project-root> --baseline <baseline.json> --trellis --trellis-user <developer-name>
```

脚本会先生成/补齐 DXM 文档；不存在 `.trellis/` 时执行 `trellis init --codex -u <developer> -y --skip-existing`，随后写入 DXM 安全块：关闭 auto-commit、让 `trellis-start` 按 selective docs 读规则、允许小修 inline，并要求完成前做对抗性检查。显式 Trellis 初始化缺命令返回 3，失败或超时返回 4，不再用 exit 0 假成功。

### 2.5 测试 / CI 模式

CI 在 Ubuntu 和 Windows 上执行：

```bash
python -m unittest discover -s tests -v
python skills/dxm/scripts/scaffold_dxm.py --self-test
```

本地发布前另外执行 `python skills/dxm/scripts/validate_dxm.py audit --root . --require-trellis`。该命令核验当前机器的 dogfood baseline/Trellis 状态，不属于干净 checkout 的 CI 步骤，因为 `.dxm/`、`.trellis/`、`.codex/` 和 `.agents/` 均是本地状态，不发布到仓库。

## 3. 配置加载链路

本仓没有运行期密钥配置。核心输入来自 CLI 参数、当前目标项目目录和仓库内模板文件。

关键来源：

- `skills/dxm/scripts/scaffold_dxm.py` 的 `FILES` 是生成文件清单的实现真相。
- `skills/dxm/scripts/dxm_contract.py` 的 schema/version/state/exit code 是 validator 契约真相。
- `skills/dxm/assets/templates/*.template` 是五份长期文档的模板来源。
- `--root` 解析为目标项目根目录；脚本会创建缺失目录，但会拒绝 broad root，除非显式传 `--allow-broad-root`。
- `--baseline` 输入必须是 UTF-8 JSON，`project_root` 必须与规范化目标 root 一致；通过后写入 `.dxm/project.json`。
- `--inventory-depth` 控制 `{{file_inventory}}` 的生成深度；默认 1。
- `--trellis-user` 默认取 `USERNAME` / `USER` / `developer`，只传给 `trellis init`。
- `--dry-run` 不写目标项目；`--refresh-blocks` 只替换管理块；`--force` 覆盖目标文件。

敏感信息边界：脚本只列文件名和职责，不读取或回显 token、密码、API Key、账号明细或密钥内容。

## 4. 核心流程

实际执行顺序：

1. **root/mode/scope lock**：agent 根据 `skills/dxm/SKILL.md` 四选一；`audit` 不进入写链路。
2. **本地证据与有界澄清**：`init` 先查证，再单批询问 0–3 个阻塞问题；结论形成 baseline。
3. **写前预检**：`scaffold_dxm.py` 校验 root 及其最近已存在祖先必须是目录、broad root、baseline schema/root、高置信凭据、既有目标 UTF-8、所有受影响 managed marker，以及受管目标/祖先不得是非目录、symlink/reparse 或多硬链接；失败时不创建文档且敏感错误只报安全字段路径，不回显凭据 key/value。
4. **模板渲染**：读取 `skills/dxm/assets/templates/`，替换反引号包裹的模板变量；validator 忽略文档中用于说明语法的 inline/fenced code token。
5. **非破坏性写入**：
   - 缺失文件：创建。
   - 已有 `AGENTS.md` 且没有 DXM block：追加 block。
   - 已有完整管理块且传 `--refresh-blocks`：仅替换管理块内部。
   - 已有人工文件且无 `--force`：跳过或追加，不静默覆盖。
6. **基线持久化（可选）**：规范化 JSON 写入 `.dxm/project.json`，专用 baseline block 幂等追加或刷新到 `项目完整链路说明.md`。
7. **Trellis 初始化与安全覆盖（可选）**：初始化失败保留普通 scaffold 产物，但返回非零；已有 `.trellis/` 时只应用安全覆盖。
8. **readiness 审计**：`audit_project()` 检查必需文件、UTF-8、真实 Markdown marker、baseline/root、未解析正文 placeholder 和可选 Trellis 完整性；完整 fenced/inline code 示例不参与 marker 计数，未闭 fence 不隐藏错误。
9. **结果输出**：区分写入动作状态与 `ABSENT/PARTIAL/READY/BROKEN`；只有 READY 才输出成功式下一步。
10. **任务完成门**：按 acceptance ID 收集 evidence 并完成对抗检查；Trellis 任务把最终 `check.md` 的文件首个非空行写成顶格独立的 `<!-- DXM-CHECK:PASS -->`，该片段全文只出现一次，任何其他或未闭合 `DXM-CHECK` 片段都失败。再依次执行 `finish`、`archive <task> --no-commit`，最后在 `.trellis/tasks/archive/<YYYY-MM>/<task>/completion.json` 生成并校验回执。CLI/file validator 同时绑定该真实输入路径、严格 `YYYY-MM` 月份目录和 check verdict，最后才作人类完成回执。普通 finish、外置回执或归档前预写 `finished: true` 都不算完成证据；receipt 会规范化 credential-like key 并向嵌套容器传播检查，除显式 env 引用/白名单脱敏占位外都拒绝且不回显。

成功信号分两层：scaffold 操作成功只证明写入完成；项目 READY 还要求 baseline、managed blocks 和请求的 Trellis 状态通过 validator。失败归因优先看 CLI 状态/issue、对应测试和 completion receipt errors。

## 5. 模块与边界

- `skills/dxm/SKILL.md`：agent 运行时契约。只描述触发、路由和操作规则，不实现文件写入。
- `skills/dxm/scripts/scaffold_dxm.py`：文件生成、marker 管理、Trellis 初始化和安全覆盖的实现层。CLI 行为以这里为准。
- `skills/dxm/scripts/dxm_contract.py`：baseline/receipt schema、marker layout、readiness 状态和退出码；不得复制一套分叉校验到 scaffold。
- `skills/dxm/scripts/validate_dxm.py`：read-only CLI adapter；不承担写入或修复。
- `skills/dxm/assets/templates/`：用户项目长期文档模板。生成到用户项目后必须自包含，不能依赖用户已安装 dxm skill。
- `skills/dxm/references/dxm-method.md`：方法论参考；只有需要解释或迁移 DXM 方法时读取。
- `skills/dxm/agents/openai.yaml`：agent 配置资产；不参与 scaffold 主流程。
- `tests/test_scaffold_dxm.py`：脚手架、baseline hydration、marker、Trellis 失败语义回归。
- `tests/test_dxm_contract.py`：baseline、audit、receipt 和 validator CLI 契约。
- `tests/test_policy_contract.py`：四模式、问题预算、evidence gate、selective docs 契约。
- `tests/test_doc_sync.py` / `tests/test_workflow_cases.py`：metadata、core-only packaging、策略资产与匿名行为案例防漂移。
- 根目录 `AGENTS.md` 与四份长期文档：本仓 dogfood DXM 的执行规范，不应替代 `skills/dxm/assets/templates/`。

## 6. 输出文件与状态流

写入用户项目的标准输出文件：

- `AGENTS.md`
- `项目开发规范（AI协作）.md`
- `项目完整链路说明.md`
- `项目文件结构说明.md`
- `开发者AI开发与PR提交流程.md`
- `.dxm/project.json`：经过 schema/root 校验的本机项目基线；使用 `--baseline` 时创建或更新。本仓 dogfood 副本含绝对 root，受 `.gitignore` 保护，不属于 release artifact；共享链路 managed block 先词法折叠路径，再使用稳定 `$PROJECT_ROOT` / `$ABSOLUTE_PATH` 投影，不泄漏或固化本机路径。

Trellis 分支可能额外影响：

- `.trellis/config.yaml`：确保 `session_auto_commit: false`。
- `.trellis/workflow.md`：追加 DXM no-task routing 与对抗性检查规则。
- `.agents/skills/trellis-start/SKILL.md`：追加 DXM Step 0。

以上 `.trellis/`、`.codex/`、`.agents/` 内容在目标项目中是工作流状态；本仓自身生成的 dogfood 副本只用于本机验证，由 `.gitignore` 排除，避免发布绝对路径、会话或任务回执。

状态含义：

- `created` / `would-create`：目标文件缺失，实际或计划创建。
- `skipped-existing` / `would-skip-existing`：已有人工文件被保护。
- `appended-dxm-block` / `appended-trellis-block`：只追加管理块。
- `appended-baseline-block`：把规范化 baseline 专用块追加到链路文档。
- `refreshed-managed-block`：只刷新 marker 内内容。
- `missing-command` / `timeout` / `failed-exit-N` / `failed-launch-<type>`：Trellis 初始化未成功；启动层 `OSError` 也必须结构化返回，不得抛 traceback。

readiness 状态：

- `ABSENT`（validator exit 4）：没有 DXM 文档或 baseline。
- `PARTIAL`（exit 3）：可继续迁移，但缺 baseline、必需文档、placeholder 或请求的 Trellis 组成部分。
- `READY`（exit 0）：baseline 与链路 managed block 一致，必需文档/marker/可选 Trellis 检查通过。
- `BROKEN`（exit 2）：JSON/root/UTF-8/marker/持久基线内容冲突等完整性错误。

scaffold 为兼容旧调用，在普通写入完成但 readiness 为 PARTIAL 时仍可 exit 0，并明确打印不具备 READY；显式 Trellis unavailable/failed/incomplete 始终非零。

## 7. 常见故障归因

| 症状 | 优先归因入口 |
| --- | --- |
| `too broad for DXM scaffold` | 目标 root 是盘根、用户根、系统目录、vendor 或构建产物；换项目根或显式 `--allow-broad-root` |
| `project root must be a directory` | `--root` 指向普通文件或其他非目录对象；改为真实项目目录 |
| `unsafe managed path` | 受管目标或祖先是非目录、symlink/reparse、多硬链接或越出锁定 root；修正路径结构后重试 |
| `not valid UTF-8` | 目标项目已有文件不是 UTF-8；先转码，脚本会避免继续写入 |
| `incomplete managed block` | DXM/Trellis marker 缺 start 或 end；修复 marker 后再跑 |
| `invalid managed block` | marker 孤立、重复或顺序错误；修复布局，不能继续追加 |
| `missing-command: trellis init --codex` / exit 3 | PATH 中没有 `trellis`；普通 scaffold 产物保留，但显式 Trellis 请求未完成 |
| `timeout` / `failed-exit-N` / `failed-launch-<type>` / `incomplete-no-trellis-dir` / exit 4 | Trellis 超时、启动/执行失败或零退出却未产出集成；检查 CLI 与 post-audit issues |
| `baseline content does not match` | `.dxm/project.json` 的可移植投影与链路 baseline managed block 漂移；用经过确认的 baseline 重新水合 |
| `contains a high-confidence credential` | baseline/receipt 某字段疑似包含真实凭据；移除或改为 redacted/env 引用，错误不会回显原值 |
| receipt validator 拒绝 | requirement/evidence kind 与 baseline 不一致，质量门失败，或 Trellis task/check/归档完成事实不可证 |
| `tests.test_doc_sync` / `test_policy_contract` 失败 | skill、模板、metadata、script 或公开文档契约漂移 |
| 发布后用户仍读到旧 skill | 未同步到实际安装目录，例如 `%USERPROFILE%\.codex\skills\dxm`，或 Codex 会话未重启 |

## DXM 大开发工作流链路

中大型开发、新模块、跨多文件重构、需求不清楚的任务，默认走：

1. 先由 DXM core 做 `project-grill`：有代码/文档时先查证，空项目按 `new-project-grill`，小脚本/demo 按 `lightweight-grill`；核心流程不依赖 sibling skill。
2. 从第一性原理出发、质疑隐藏假设，先从代码和文档自行判断，再单批提出 0–3 个阻塞问题；`grill-with-docs` 可在已安装且任务描述匹配时做同样有界的查证，full `grilling` / legacy `grill-me` 仅在用户 explicit opt-in 深度澄清时作为 optional 增强。
3. 把结论写入 `.trellis/tasks/<task>/prd.md`，不能只停留在聊天上下文里。
4. 用 `.trellis/scripts/task.py start <task>` 进入 Trellis active task。
5. 按 Trellis 的 implement/check/update-spec 节奏开发、验证、沉淀规范。
6. 任务完成后执行对抗性检查，挑战需求偏差、隐藏假设、负路径、架构边界、测试、文档、敏感信息、乱码和回滚/恢复。
7. 对抗性检查通过后同步 DXM 长期文档；不能只更新 `.trellis/` 内部状态。
8. 先把最终 `check.md` 的文件首个非空行写成顶格独立的 `<!-- DXM-CHECK:PASS -->`，确保该片段全文唯一且没有其他或未闭合 `DXM-CHECK` 片段；再执行 `finish` 和 `archive <task> --no-commit`，在 `.trellis/tasks/archive/<YYYY-MM>/<task>/completion.json` 生成并校验回执；归档前不得预写 `finished: true`。

## DXM 项目基线

- Schema version: `1`
- Project root: local canonical path stored in `.dxm/project.json`; not embedded in shared Markdown.
- Goal: Provide a deterministic, evidence-backed DXM workflow that separates scaffold completion from project readiness.

### Primary users / callers

- DXM skill maintainers
- agents and developers governing medium-to-large projects

### Deliverables

- self-contained core DXM skill
- safe scaffold and read-only validator
- governance templates and behavior regression tests

### Non-goals

- automatic Git commit, push, PR, tag, or release
- mandatory exhaustive grilling
- claiming live or UI acceptance from source-only evidence

### Runtime entry points

- $dxm
- python skills/dxm/scripts/scaffold_dxm.py
- python skills/dxm/scripts/validate_dxm.py

### Runtime facts

- Python standard library only for deterministic tooling
- sibling interview skills are optional enhancements
- Trellis is optional and requires explicit approval or request

### Acceptance criteria

| ID | Criterion | Required evidence |
| --- | --- | --- |
| AC-01 | Route work through exactly one of audit, init, task, or scaffold-only. | policy-contract, workflow-case |
