## 1. 项目目标

`dxm` 是 DXM 大项目 AI 协作 skill 的源码仓。它把“首次建档、project-grill 澄清、长期文档同步、Trellis 大任务治理、发布完成面”沉淀为可安装 skill 与脚手架，供 Codex、Claude Code 或其他支持 Agent Skills 的运行时复用。

主要调用方：

- 触发 `/dxm` 或 `/dxm trellis` 的 agent：先读 `skills/dxm/SKILL.md`，再根据目标目录状态执行 project-grill 或 scaffold。
- 维护者：直接运行 `skills/dxm/scripts/scaffold_dxm.py`，验证模板输出、Trellis 集成和安装包冒烟。

成功产物：

- `skills/dxm/SKILL.md`：运行时触发契约。
- `skills/dxm/scripts/scaffold_dxm.py`：实际 scaffold / dry-run / refresh / Trellis 初始化实现。
- `skills/dxm/assets/templates/`：写入用户项目的五份长期文档模板。
- `skills/grilling`、`skills/grill-with-docs`、`skills/grill-me`、`skills/domain-modeling`：DXM 路由依赖的协作技能。

## 2. 运行模式

### 2.1 Skill 触发模式

用户在某个项目目录输入 `/dxm`、`/dxm trellis` 或要求“生成/刷新 DXM 大项目开发规范”时，agent 读取 `skills/dxm/SKILL.md`：

1. 判断目标根目录，避免误在盘根、用户根、vendor、构建产物或叶子目录初始化。
2. 识别 `只分析 / 先看看`、`scaffold only / 先别问`、已有完整 DXM、空目录、已有代码/文档等分支。
3. 对需要澄清的分支，先应用第一性原理和质疑规则，再路由到 `grilling`、`grill-with-docs`、`domain-modeling` 或 legacy `grill-me`。
4. 意图清楚后运行 `skills/dxm/scripts/scaffold_dxm.py`。

### 2.2 Scaffold CLI 模式

维护者或 agent 可直接运行：

```bash
python skills/dxm/scripts/scaffold_dxm.py --root <project-root>
```

常用变体：

- `--dry-run`：只报告计划动作，不写文件。
- `--refresh-blocks`：只刷新 DXM/Trellis 管理块，保留人工内容。
- `--inventory-depth N`：控制初始化文件结构种子的目录深度。
- `--force`：覆盖既有 DXM 目标文件；只允许在用户明确接受丢失人工内容风险时使用。
- `--self-test`：对已安装 skill 做轻量冒烟。

### 2.3 Trellis 模式

运行：

```bash
python skills/dxm/scripts/scaffold_dxm.py --root <project-root> --trellis --trellis-user <developer-name>
```

脚本会先生成/补齐 DXM 文档，再执行 `trellis init --codex -u <developer> -y --skip-existing`，随后写入 DXM 的 Trellis 安全块：关闭 auto-commit、让 `trellis-start` 先读 DXM 文档、允许小修 inline，并要求完成前做对抗性检查。

### 2.4 测试 / CI 模式

CI 在 Ubuntu 和 Windows 上执行：

```bash
python -m unittest discover -s tests -v
python skills/dxm/scripts/scaffold_dxm.py --self-test
```

## 3. 配置加载链路

本仓没有运行期密钥配置。核心输入来自 CLI 参数、当前目标项目目录和仓库内模板文件。

关键来源：

- `skills/dxm/scripts/scaffold_dxm.py` 的 `FILES` 是生成文件清单的实现真相。
- `skills/dxm/assets/templates/*.template` 是五份长期文档的模板来源。
- `--root` 解析为目标项目根目录；脚本会创建缺失目录，但会拒绝 broad root，除非显式传 `--allow-broad-root`。
- `--inventory-depth` 控制 `{{file_inventory}}` 的生成深度；默认 1。
- `--trellis-user` 默认取 `USERNAME` / `USER` / `developer`，只传给 `trellis init`。
- `--dry-run` 不写目标项目；`--refresh-blocks` 只替换管理块；`--force` 覆盖目标文件。

敏感信息边界：脚本只列文件名和职责，不读取或回显 token、密码、API Key、账号明细或密钥内容。

## 4. 核心流程

实际执行顺序：

1. **触发与分类**：agent 根据 `skills/dxm/SKILL.md` 判断 read-only、scaffold-only、new-project、existing-docs、Trellis 等分支。
2. **预检**：`scaffold_dxm.py` 校验 broad root、UTF-8、管理块 marker 成对性；发现非 UTF-8 或残缺 marker 时失败并避免继续写入。
3. **模板渲染**：读取 `skills/dxm/assets/templates/`，替换 `{{project_name}}`、`{{project_root}}`、`{{generated_date}}`、`{{file_inventory}}`。
4. **非破坏性写入**：
   - 缺失文件：创建。
   - 已有 `AGENTS.md` 且没有 DXM block：追加 block。
   - 已有完整管理块且传 `--refresh-blocks`：仅替换管理块内部。
   - 已有人工文件且无 `--force`：跳过或追加，不静默覆盖。
5. **Trellis 初始化（可选）**：`--trellis` 分支调用 `trellis init --codex -u <developer> -y --skip-existing`；缺少 `trellis` 时报告 `missing-command`，但保留普通 DXM scaffold 结果。
6. **Trellis 安全覆盖（可选）**：当 `.trellis/` 存在时，写入或刷新 AGENTS / 长期文档 / `.trellis/config.yaml` / `trellis-start` / workflow 的 DXM 管理块。
7. **结果输出**：CLI 打印 root、created/skipped/refreshed/missing-command/timeout 等状态摘要，测试据此断言行为。

成功信号：目标目录存在五份 DXM 文档；CLI 退出码为 0；相关测试通过；Trellis 分支额外出现 `.trellis/` 和安全块。失败归因优先看 CLI stderr/stdout 与 `tests/test_scaffold_dxm.py` 对应断言。

## 5. 模块与边界

- `skills/dxm/SKILL.md`：agent 运行时契约。只描述触发、路由和操作规则，不实现文件写入。
- `skills/dxm/scripts/scaffold_dxm.py`：文件生成、marker 管理、Trellis 初始化和安全覆盖的实现层。CLI 行为以这里为准。
- `skills/dxm/assets/templates/`：用户项目长期文档模板。生成到用户项目后必须自包含，不能依赖用户已安装 dxm skill。
- `skills/dxm/references/dxm-method.md`：方法论参考；只有需要解释或迁移 DXM 方法时读取。
- `skills/dxm/agents/openai.yaml`：agent 配置资产；不参与 scaffold 主流程。
- `tests/test_scaffold_dxm.py`：脚手架行为回归测试。
- `tests/test_doc_sync.py`：防漂移 tripwire，约束 `SKILL.md`、模板和脚本关键契约一致。
- 根目录 `AGENTS.md` 与四份长期文档：本仓 dogfood DXM 的执行规范，不应替代 `skills/dxm/assets/templates/`。

## 6. 输出文件与状态流

写入用户项目的标准输出文件：

- `AGENTS.md`
- `项目开发规范（AI协作）.md`
- `项目完整链路说明.md`
- `项目文件结构说明.md`
- `开发者AI开发与PR提交流程.md`

Trellis 分支可能额外影响：

- `.trellis/config.yaml`：确保 `session_auto_commit: false`。
- `.trellis/workflow.md`：追加 DXM no-task routing 与对抗性检查规则。
- `.agents/skills/trellis-start/SKILL.md`：追加 DXM Step 0。

状态含义：

- `created` / `would-create`：目标文件缺失，实际或计划创建。
- `skipped-existing` / `would-skip-existing`：已有人工文件被保护。
- `appended-dxm-block` / `appended-trellis-block`：只追加管理块。
- `refreshed-managed-block`：只刷新 marker 内内容。
- `missing-command` / `timeout` / `failed-exit-N`：Trellis 初始化未成功。

## 7. 常见故障归因

| 症状 | 优先归因入口 |
| --- | --- |
| `too broad for DXM scaffold` | 目标 root 是盘根、用户根、系统目录、vendor 或构建产物；换项目根或显式 `--allow-broad-root` |
| `not valid UTF-8` | 目标项目已有文件不是 UTF-8；先转码，脚本会避免继续写入 |
| `incomplete managed block` | DXM/Trellis marker 缺 start 或 end；修复 marker 后再跑 |
| `missing-command: trellis init --codex` | PATH 中没有 `trellis`；普通 DXM 文档已可用，Trellis 初始化被跳过 |
| `timeout: trellis init --codex` | Trellis 初始化超时；检查 `--trellis-timeout-seconds`、PATH、Trellis CLI 状态 |
| `tests.test_doc_sync` 失败 | `skills/dxm/SKILL.md`、`assets/templates/`、`scaffold_dxm.py` 三处契约漂移 |
| 发布后用户仍读到旧 skill | 未同步到实际安装目录，例如 `C:\Users\admin\.codex\skills\dxm`，或 Codex 会话未重启 |
